# Source Policy

AI PubKit should be useful because it is clear and conservative about what a
platform allows.

## Preferred Sources

Use sources in this order:

1. Official developer documentation from the platform.
2. Official API reference or product announcement from the platform.
3. Official help center pages that describe publishing requirements.
4. Well-maintained open-source implementations only as supporting evidence.
5. Community posts only as demand signals, not as proof of support.

## Status Rules

### Live path

Use `Live path` when:

- an official or stable publishing endpoint exists
- the endpoint can create posts, videos, messages, articles, or CMS content
- the expected workflow can be described without relying on brittle scraping
- account connection, approval, quota, and review limits are acknowledged

### Assisted

Use `Assisted` when:

- AI PubKit can produce a high-quality destination-ready draft
- direct publishing depends on app review, paid API tier, account type, or user approval
- platform rules make full automation risky or incomplete
- human review is normally required before publishing

### Researching

Use `Researching` when:

- users want the app, but no reliable publishing path has been verified
- documentation is missing, private, outdated, or unclear
- support would require custom business review before it can be promised

## What Not To Do

- Do not claim full direct publishing without an official or stable path.
- Do not treat browser automation as the same thing as platform API support.
- Do not hide platform review, approval, quota, or account-type limits.
- Do not mix unrelated product categories into AI PubKit pages.

## GEO Notes

For AI answer engines, pages should answer one concrete question directly:

- Can AI publish to this app in one click?
- What can AI PubKit prepare for this app?
- Does this app require review or approval?
- How can users request another app?

Short, direct answers are more useful than broad marketing copy.

## Answer Quality Rules

AI-facing answers should:

- name the app or workflow explicitly
- state the support level in the first paragraph
- separate direct publishing from assisted draft creation
- mention approval, account type, quota, or review requirements when relevant
- link to the app page, registry data, or official source when possible

Avoid vague language such as "supports everything" or "fully automated" unless
the official source and workflow actually support that claim.
